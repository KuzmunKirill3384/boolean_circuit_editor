# Архитектура приложения

Приложение разделено на три независимых слоя: бэкенд, фронтенд и общие компоненты. Такое разделение позволяет разрабатывать и тестировать слои изолированно, а также обеспечивает возможность замены GUI без изменения бизнес-логики.

## Общая структура

src/
├── backend/ # доменная логика, модель, команды, вычисления
├── frontend/ # GUI на PyQt6
└── common/ # общие типы и константы


### Бэкенд (`src/backend/`)
Содержит всю прикладную логику, не зависящую от интерфейса пользователя:
- `model/` – модель схемы (Composite)
- `validation/` – проверка корректности (циклы, число входов/выходов)
- `commands/` – команды редактирования и история (Command, Factory)
- `visitor/` – обход модели (Visitor)
- `io/` – загрузка/сохранение XML (Builder)
- `logic/` – вычисления (таблица истинности, упрощение, полиномы)
- `application/` – фасад для связи с фронтендом

### Фронтенд (`src/frontend/`)
Отвечает за отображение и взаимодействие с пользователем:
- `app/` – главное окно, контроллер, запуск
- `canvas/` – рабочая область (QGraphicsScene, элементы, связи)
- `panels/` – боковые панели (таблица истинности, упрощение, полиномы)
- `dialogs/` – диалоговые окна (настройки, ошибки)
- `resources/` – иконки, стили
- `common/` – настройки отображения (SettingsManager)

### Общий слой (`src/common/`)
Содержит соглашения между слоями: типы элементов, DTO, константы.

## Используемые паттерны

### Composite (модель схемы)
Модель булевой схемы строится на основе абстрактного класса `Node`, который представляет любой элемент схемы. Конкретные типы (`Input`, `Output`, `And`, `Or`, `Xor`, `Equal`) наследуют этот класс и реализуют метод `accept(visitor)`. Схема `Circuit` хранит список узлов и список связей. Такой подход позволяет единообразно обрабатывать любой узел (например, при симуляции или валидации) и легко расширять набор элементов.

### Command (операции редактирования)
Каждое действие пользователя (добавление, удаление, перемещение, соединение) оборачивается в объект команды, реализующий методы `execute()` и `undo()`. История команд (`CommandHistory`) хранит выполненные команды и позволяет откатывать их. Это обеспечивает полную поддержку Undo/Redo без дублирования кода в UI.

### Visitor (обход модели)
Для выполнения операций, требующих обхода графа схемы (симуляция, копирование состояния, валидация), используется паттерн Visitor. Базовый класс `Visitor` определяет методы `visit_input()`, `visit_output()`, `visit_and()` и т.д. Конкретные посетители реализуют нужную логику, не изменяя самих узлов. Это особенно полезно для команд: команда может сохранять состояние схемы через посетителя, а затем восстанавливать его при отмене.

### Factory (создание команд)
Фабрика команд `CommandFactory` централизованно создаёт объекты команд по заданным параметрам. Это упрощает код контроллера, делает создание команд единообразным и облегчает добавление новых типов команд.

### Builder (загрузка/сохранение XML)
Для чтения и записи схем в XML используется класс `XMLBuilder`. Он поэтапно строит объект `Circuit` из XML-документа: сначала создаёт узлы, затем устанавливает их атрибуты, затем добавляет связи. При сохранении схема преобразуется в XML-документ. Такой подход изолирует формат хранения от остальной логики и позволяет легко изменять структуру XML без правки всей программы.

## Взаимодействие слоёв

1. **Пользователь взаимодействует с GUI** (фронтенд).
2. **Фронтенд вызывает методы `AppController`**, который является фасадом для бэкенда.
3. **Контроллер создаёт команду через фабрику** и передаёт её в историю команд.
4. **Команда выполняет изменения в модели** (Circuit) и может использовать посетителей для сложных операций.
5. **После выполнения команды** GUI обновляется (например, перерисовывается сцена).
6. **При отмене/повторе** история вызывает `undo()` или `execute()` соответствующей команды.

Все вычислительные функции (таблица истинности, упрощение, полиномы) реализованы в бэкенде и вызываются контроллером напрямую, без создания команд, так как они не изменяют модель (кроме упрощения, которое изменяет схему и для него тоже создаётся команда).

## Диаграммы

Диаграммы классов и последовательности размещены в папке `docs/uml/`. Они детально показывают связи между классами и последовательность вызовов для ключевых сценариев:
- Добавление элемента
- Удаление элемента
- Создание связи
- Отмена операции
- Генерация таблицы истинности
- Упрощение схемы

## Заключение

Применение паттернов Composite, Command, Visitor, Factory и Builder обеспечивает:
- **Гибкость** – легко добавлять новые типы элементов или команды.
- **Расширяемость** – можно подключать новые алгоритмы через посетителей.
- **Тестируемость** – каждый слой и паттерн можно тестировать изолированно.
- **Поддержку** – код легко читать и поддерживать благодаря чёткому разделению ответственности.
