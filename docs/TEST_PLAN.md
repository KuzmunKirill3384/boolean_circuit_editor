# План тестирования

Этот документ нужен, чтобы тестирование не появилось в самом конце проекта. За ведение плана отвечает **Боос Д. А.**

## Цели

- проверить корректность модели схемы;
- проверить устойчивость к некорректным связям;
- проверить работу undo/redo;
- проверить загрузку и сохранение XML;
- проверить вычислительные функции;
- проверить основные пользовательские сценарии GUI.

## Уровни тестирования

### Unit

Проверяются отдельные модули:

- элементы схемы;
- валидация;
- команды;
- XML Builder;
- логика таблицы истинности;
- логика упрощения;
- полиномиальное представление.

### Integration

Проверяются сценарии на стыке модулей:

- команда изменяет модель и корректно откатывается;
- схема сохраняется в XML и затем восстанавливается без потери структуры;
- фронтенд корректно получает данные из backend facade.

### Manual GUI

Ручные сценарии:

- создать схему с нуля;
- соединить элементы;
- получить ошибку при некорректной связи;
- откатить и повторить действие;
- открыть таблицу истинности;
- упростить схему;
- сохранить и загрузить XML.

## Набор тестовых схем

В `tests/fixtures/` должны появиться:

- минимальная схема из одного входа и одного выхода;
- схема с двумя входами и одним `AND`;
- схема с несколькими логическими элементами;
- схема для проверки упрощения;
- набор некорректных XML-файлов.

## Критерии готовности

- все unit-тесты проходят;
- все integration-тесты проходят;
- ручной чек-лист GUI закрыт;
- ключевые ошибки формата XML покрыты тестами.
